**BUG-04 Fix: Wrap ws.send() errors in ConnectionError per D-08 from CONTEXT.md**

Two changes required:

1. **Add CONNECTION_SEND_FAILED to ConnectionErrorCode** in src/errors.ts:
```typescript
export type ConnectionErrorCode =
  | 'TLS_FINGERPRINT_MISMATCH'
  | 'CONNECTION_STALE'
  | 'CONNECTION_CLOSED'
  | 'CONNECT_TIMEOUT'
  | 'CONNECTION_REFUSED'
  | 'NETWORK_ERROR'
  | 'PROTOCOL_ERROR'
  | 'CONNECTION_SEND_FAILED';  // ADD THIS
```

2. **In websocket.ts send() method (lines 416-432)**, change the catch block:

Current (throws raw error):
```typescript
} catch (err) {
  const error: WebSocketError = { ... };
  this.handleError(error);
  throw error;  // throws WebSocketError, not ConnectionError
}
```

Change to:
```typescript
} catch (err) {
  const wsError: WebSocketError = {
    message: `Failed to send data: ${err instanceof Error ? err.message : String(err)}`,
    original: err instanceof Error ? err : undefined,
    recoverable: false,
  };
  this.handleError(wsError);

  // Throw typed ConnectionError instead of raw WebSocketError
  const connError = new ConnectionError({
    code: 'CONNECTION_SEND_FAILED',
    message: wsError.message,
    retryable: false,
    details: { originalError: err instanceof Error ? err.message : String(err) },
  });
  throw connError;
}
```

Note: Keep `this.handleError(error)` call before throwing — per D-08 in CONTEXT.md, this preserves the existing diagnostic behavior (onerror callback still fires).

**BUG-05 Fix: Return silently on duplicate response per D-09, D-10 from CONTEXT.md**

In `resolveRequest()` method (lines 95-112), change the error-throwing case to a debug return:

Current (throws on duplicate):
```typescript
resolveRequest(id: string, response: ResponseFrame): void {
  const entry = this.pendingRequests.get(id);

  if (!entry) {
    throw new Error(`No pending request with ID "${id}"`);  // BUG: throws on duplicate
  }
  // ...
}
```

Change to:
```typescript
resolveRequest(id: string, response: ResponseFrame): void {
  const entry = this.pendingRequests.get(id);

  if (!entry) {
    // Duplicate server response — this is expected in retry scenarios
    // Log at debug level and return silently without throwing
    return;
  }
  // ...
}
```

**Note:** The RequestManager class does not currently have a logger. Add a private logger property similar to EventManager's pattern:

```typescript
import { type Logger, LogLevel } from '../types/logger.js';

private logger: Logger = {
  name: 'request-manager',
  level: LogLevel.Error,
  debug() {},
  info() {},
  warn() {},
  error(message: string, meta?: Record<string, unknown>) {
    console.error(message, meta ?? '');
  },
};
```

Then in `resolveRequest()`, before the `return;` for duplicate case, add:
```typescript
this.logger.debug(`Duplicate response received for request ID "${id}" — ignoring`);
```

This is DEBUG level per BUG-05 specifics in CONTEXT.md — these are expected in retry scenarios and should not appear in normal logs.
